# litert-lm-server

A small, **standalone** OpenAI-compatible HTTP server wrapping Google's
[LiteRT-LM](https://github.com/google-ai-edge/LiteRT-LM) C ABI. It loads one
`.litertlm` model at startup and serves chat completions generated by LiteRT on
the **CPU** backend.

- **lemonade-AUTHORED** — this engine wrapper is written and maintained by the
  Lemonade project. Lemonade launches it as a subprocess backend (à la
  `llama-server`, `whisper-server`, etc.).
- **GitHub-hosted** — developed at `github.com/ianbmacdonald/litert-lm-server`,
  following Lemonade's standalone subprocess-server pattern (e.g. `ort-server`);
  intended for `lemonade-sdk` upstream once an MVP is reviewed. The prpl OpenWrt
  feed consumes it from here by pinned commit; prpl owns the buildroot package
  recipe, Lemonade owns this source.

It links against the prebuilt `liblitert-lm.so` and does **not** pull in
LiteRT's Bazel build system. Only two vendored single-header libraries are used
(`third_party/httplib.h`, `third_party/json.hpp`).

## Build

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j
```

The path to the LiteRT-LM build tree is baked into `CMakeLists.txt` via
`LITERT_LM_ROOT` (default `/home/imac/lemonade-litert-build/bazel-native/LiteRT-LM`).
Override it if the tree moves:

```bash
cmake -S . -B build -DLITERT_LM_ROOT=/path/to/LiteRT-LM
```

### How it links the `.so`

- Header: `${LITERT_LM_ROOT}/c/engine.h`
- Library: `${LITERT_LM_ROOT}/bazel-bin/c/liblitert-lm.so` (self-contained —
  resolves every engine symbol; no `libengine_cpu.a` needed).
- The library directory is embedded as an **rpath** (`BUILD_RPATH` /
  `INSTALL_RPATH`), so the binary runs without setting `LD_LIBRARY_PATH`.

Equivalent one-liner:

```bash
g++ -std=c++17 -O2 server.cpp \
  -I "$LITERT_LM_ROOT/c" -I third_party \
  -L "$LITERT_LM_ROOT/bazel-bin/c" -llitert-lm \
  -Wl,-rpath,"$LITERT_LM_ROOT/bazel-bin/c" -lpthread \
  -o litert-lm-server
```

## Run

```bash
./build/litert-lm-server --model /path/to/model.litertlm --port 8080 \
    [--host 0.0.0.0] [--model-id my-model]
```

Engine creation is heavy and happens once at startup (a few seconds). `/health`
returns 200 only after the engine is ready.

## Endpoints (OpenAI-compatible)

| Method | Path | Notes |
|--------|------|-------|
| GET  | `/health` | `{"status":"ok"}` |
| GET  | `/v1/models`, `/models` | the one loaded model, OpenAI list shape |
| POST | `/v1/chat/completions`, `/chat/completions` | non-streaming and `stream:true` SSE |

Accepted request fields: `model` (ignored — one model per process),
`messages:[{role,content}]` (content may be a string or an OpenAI parts array),
`stream`, `temperature`, `max_tokens` / `max_completion_tokens`. System messages
and multi-turn history are supported.

## Notes for a Lemonade WrappedServer backend author

- **Contract**: poll `GET /health` for readiness, then `POST /v1/chat/completions`.
  Both non-streaming and SSE streaming responses follow the OpenAI schema
  (`chat.completion` / `chat.completion.chunk`, terminated by `data: [DONE]`).
- **One model per process** — the engine is bound to a single model at startup.
  Run one process per model; there is no runtime load/unload.
- **Serialized inference** — a single global mutex serializes all generation
  through the one engine (matches Lemonade's router model). Concurrent requests
  queue; they do not run in parallel.
- **CPU backend only** in this build. The engine is created with backend `"cpu"`.
- **Sampler**: this LiteRT-LM build implements **only TopP** sampling
  (`kLiteRtLmSamplerTypeTopP`). `Greedy` and `TopK` return
  `UNIMPLEMENTED`, so the server always uses TopP (top_k=40, top_p=0.95) with the
  request's `temperature` (clamped to a small positive minimum). Do not expect
  deterministic greedy decoding from this engine build.
- **Model output** includes the model's `<think>...</think>` reasoning span
  verbatim in `message.content` (Qwen3 emits it); strip it downstream if unwanted.
- **Usage** counts are computed with the engine tokenizer
  (`litert_lm_engine_tokenize`) over the input and output text.
